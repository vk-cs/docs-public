VK Object Storage позволяет настроить жизненный цикл (lifecycle) объектов в бакете. Жизненный цикл — это автоматизированное удаление из бакета объектов или их [версий](../versioning) по заданным правилам.

Правила устанавливаются на уровне одного бакета. Настроить жизненный цикл для группы бакетов или проекта нельзя.

[В личном кабинете](../../instructions/manage_lifecycle) можно задать каждое правило по отдельности, а с помощью [API](/ru/tools-for-using-services/api/api-spec/s3-rest-api/lifecycle-api) и [AWS CLI](../../instructions/manage-lifecycle) — настроить конфигурацию правил жизненного цикла, передав в VK Object Storage все правила в одном файле. Для API файл подготавливается в формате XML, для AWS CLI — в формате JSON.

При добавлении правила оно автоматически применяется к соответствующим объектам или версиям объектов, которые затем удаляются в фоновом режиме: при срабатывании правила они становятся недоступны, а затем удаляются в порядке очереди в течение следующих 24 часов.

Ограничения:

- Для каждого бакета можно установить не более 50 правил.
- При создании правил через API размер XML-файла с правилами не должен превышать 512 КБ.

## {heading(Правила жизненного цикла в бакетах без версионирования)[id=lifecycle_rules]}

Правила жизненного цикла содержат следующие элементы:

- `ID` — уникальный идентификатор правила.
- `Status` — статус правила. Возможные значения:

  - `Enabled` — правило применяется.
  - `Disabled` — правило не применяется.

- `Filter` — фильтр для выбора объектов.

  Поддерживается фильтрация по [префиксу](/ru/storage/s3/concepts/about#object_key) ключа объекта (параметр `Prefix`). Примеры префиксов для ключа: `image`, `image/`, `image/photo/`. Фильтр может содержать только один префикс. Для каждого префикса необходимо создавать отдельное правило.

- `Expiration` — время жизни объекта. Задается с помощью параметров:

  - `Days` — количество дней после загрузки в бакет, через которое объект должен быть удален.
  - `Date` — дата, когда объект должен быть удален. Можно указать только с помощью API и AWS CLI.

### {heading(Пример JSON-файла конфигурации для бакета без версионирования)[id=lifecycle_config]}

В примере приведено правило, согласно которому все объекты в директории бакета `tmp` будут удаляться через день после их загрузки в бакет.

```json
{
  "Rules": [
    {
      "ID": "Object expiration rule",
      "Status": "Enabled",
      "Expiration": {
        "Days": 1
      },
      "Filter": {
        "Prefix": "tmp/"
      }
    }
  ]
}
```

## {heading(Правила жизненного цикла в бакетах c версионированием)[id=lifecycle_rules_with_versioning]}

Правила жизненного цикла в бакетах с [версионированием](../versioning) определяют порядок автоматизированного удаления из бакета текущих и неактуальных версий объектов и маркеров удаления.

Основные элементы для построения правил:

- `ID` — уникальный идентификатор правила.
- `Status` — статус правила. Возможные значения:

  - `Enabled` — правило применяется.
  - `Disabled` — правило не применяется.

- `Filter` — фильтр для выбора объектов.

  Поддерживается фильтрация по [префиксу](/ru/storage/s3/concepts/about#object_key) ключа объекта (параметр `Prefix`). Примеры префиксов для ключа: `image`, `image/`, `image/photo/`. Фильтр может содержать только один префикс. Для каждого префикса необходимо создавать отдельное правило.

- `Expiration` — условия удаления текущей версии объекта. Задаются с помощью параметров:

  - `Days` — количество дней после загрузки текущей версии объекта в бакет, через которое она должна быть удалена.
  - `Date` — дата, когда должна быть удалена текущая версия объекта. Можно указать только с помощью API и AWS CLI.
  - `ExpiredObjectDeleteMarker` — определяет, нужно ли удалять из бакета маркер удаления объекта, когда других версий этого объекта не осталось. Принимает значения `true` и `false`.

- `NoncurrentVersionExpiration` — условия удаления неактуальных версий объекта. Задаются с помощью параметров:

  - `NoncurrentDays` — максимальное количество дней, в течение которых версия объекта неактуальна. По истечении этого времени неактуальная версия будет удалена.
  - `NewerNoncurrentVersions` — максимальное количество неактуальных версий (от 1 до 100), которые нужно сохранять. Когда их станет больше, самые старые версии будут удалены. Вместе с этим параметром в правиле также должен быть указан параметр `Filter`. В противном случае VK Object Storage выдаст ошибку `InvalidRequest`.

  Если в правиле указаны оба этих параметра, должны быть превышены значения и `NoncurrentDays`, и `NewerNoncurrentVersions`, чтобы произошло удаление неактуальной версии.

### Применение правил жизненного цикла к версиям объекта

Результат применения правил зависит от [статуса версионирования](../versioning#versioning_status) бакета и наличия [маркера удаления](../versioning#delete_marker) объекта:

- Если текущая версия — это маркер удаления, проверяется наличие других версий.

  - Если нет других версий, объект будет удален безвозвратно.
  - Если есть другие версии, маркер не будет удален. Существующие версии также останутся без изменения.

- Если включено версионирование и текущая версия имеет [уникальный идентификатор](../versioning#version_id), будет создан маркер удаления, который станет текущей версией объекта.
- Если версионирование приостановлено, будет удалена версия объекта с идентификатором `null`:

  - Если нет других версий, ее удаление приведет к безвозвратному удалению объекта из бакета.
  - Если есть другие версии, будет создан маркер удаления с идентификатором `null`, который станет текущей версией объекта.

### Пример JSON-файла конфигурации для бакета с версионированием

В примере приведен набор из трех правил жизненного цикла. Согласно этому набору:

- Текущие версии всех объектов в директории `tmp` бакета будут удаляться через 30 дней после их загрузки в бакет.
- Для всех объектов в бакете:

  - Будет удаляться самая старая версия объекта, неактуальная более трех дней, если неактуальных версий объекта больше пяти.
  - Будут удаляться маркеры удаления, которые являются единственными версиями объектов.

```json
{
  "Rules": [
    {
      "ID": "Current version expiration rule",
      "Status": "Enabled",
      "Expiration": {
        "Days": 30
      },      
      "Filter": {
        "Prefix": "tmp/"
      }
    },        
    {
      "ID": "Noncurrent version expiration rule",
      "Status": "Enabled",
      "Filter": {
      },
      "NoncurrentVersionExpiration": {
        "NewerNoncurrentVersions": 5,
        "NoncurrentDays": 3
      }
    },
    {
      "ID": "Single delete marker expiration rule",
      "Status": "Enabled",
      "Expiration": {
        "ExpiredObjectDeleteMarker": true
      },
    }     
  ]
}
```
