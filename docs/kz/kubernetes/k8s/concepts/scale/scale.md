# {heading(Кластер түйіндерін масштабтау)[id=k8s-scale]}

{include(/kz/_includes/_translated_by_ai.md)}

Масштабтау кластерді жұмыс жүктемелерінің ([workloads](https://kubernetes.io/docs/concepts/workloads/)) өзгеретін қажеттіліктеріне бейімдеуге мүмкіндік береді.

## {heading(Масштабтау түрлері)[id=k8s-scale-types]}

Масштабтаудың екі түрі қолданылады:

- _Тік масштабтау_: кластердің master- және worker-түйіндері пайдаланатын {linkto(../flavors#k8s-flavors)[text=виртуалды машина қалыптары]} өзгереді. Осылайша түйіндер санына әсер етпей, кластердің есептеу ресурстарын басқаруға болады.

  Worker-түйіндерді масштабтау {linkto(../architecture#k8s-architecture-topology)[text=worker-түйіндер тобы]} аясында {linkto(../cluster-autoscaler#k8s-cluster-autoscaler)[text=Cluster Autoscaler]} көмегімен орындалады.

- _Көлденең масштабтау_: жекелеген түйіндер тобы аясындағы worker-түйіндер саны өзгереді. Осылайша түйіндер пайдаланатын виртуалды машина қалыптарына әсер етпей, кластердің есептеу ресурстарын басқаруға болады.

Екі түрдегі масштабтауды да қолмен орындауға болады, сондай-ақ {linkto(#k8s-scale-autoscaling)[text=автоматты масштабтау]} жұмыс істейді.

## {heading(Автоматты масштабтау)[id=k8s-scale-autoscaling]}

### {heading(Тік масштабтау)[id=k8s-scale-vertical-autoscaling]}

Master-түйіндердің _тік автоматты масштабталуы_ барлық кластерлер үшін қолданылады, оны өшіруге болмайды.

Автоматты масштабтау агенті CPU және RAM бойынша master-түйін жүктемесін бағалап, келесі шектік мәндерді қадағалайды:

[cols="1,2", options="header"]
|===

|Шектік мән
|Автомасштабтаудың әрекеті

|CPU жүктемесі 60 секунд бойы 80%-дан асады
.2+|Егер шектердің кемінде бірі асса, Cloud Containers сервисіне master-түйіннің ВМ қалыбын өзгерту туралы сұрау жіберіледі:

- CPU буыны өзгермейді.
- CPU ядроларының саны екіге артады.
- RAM көлемі тек CPU санын арттыру үшін қажет болған жағдайда ғана өзгереді.

Мысалдар:

- `STD2-2-6` → `STD2-4-6`.
- `STD2-6-6` → `STD2-8-8`.
- `STD3-2-6` → `STD3-4-6`.

|CPU жүктемесі 5 минут бойы 60%-дан асады

|RAM жүктемесі 60 секунд бойы 90%-дан асады
|Cloud Containers сервисіне master-түйіннің ВМ қалыбын өзгерту туралы сұрау жіберіледі:

- CPU буыны өзгермейді.
- CPU ядроларының саны тек RAM көлемін арттыру үшін қажет болған жағдайда ғана өзгереді.
- RAM көлемі келесі ретпен бір қадамға өзгереді: 6, 8, 12, 16, 20, 24, 32, 36, 48.

Мысалдар:

- `STD2-2-6` → `STD2-2-8`.
- `STD2-2-8` → `STD2-4-12`.
- `STD3-2-6` → `STD3-2-8`.
|===

Master-түйіннің ВМ қалыбын CPU мен RAM мөлшері аз қалыпқа тек {linkto(../../instructions/scale#k8s-instructions-scale-vertical-master-nodes)[text=қолмен]}.

{cut(06.12.2024 дейін жасалған кластерлер үшін тік масштабтау сұлбасы)}

Автоматты масштабтау агенті CPU және RAM бойынша master-түйін жүктемесін бағалап, келесі шектік мәндерді қадағалайды:

- CPU жүктемесі 60 секунд бойы 80%-дан асады;
- CPU жүктемесі 5 минут бойы 60%-дан асады;
- RAM жүктемесі 60 секунд бойы 90%-дан асады.

Егер шектердің кемінде бірі асса, Cloud Containers сервисіне master-түйіннің ВМ қалыбын өзгерту туралы сұрау жіберіледі. Бұл ретте CPU және RAM мәндері екі есе артады. Мысалы, `STD2-2-6` қалыбы `STD2-4-12` болып өзгереді.

{/cut}

### {heading(Көлденең масштабтау)[id=k8s-scale-horizontal-autoscaling]}

Түйіндер тобы ({linkto(../cluster-autoscaler#k8s-cluster-autoscaler)[text=autoscaling]}) үшін _көлденең автоматты масштабтау_ пайдаланушы арқылы {linkto(../../instructions/scale#k8s-instructions-scale-horizontal-autoscaling-worker-nodes)[text=бапталады]}. Топтағы worker-түйіндер саны жұмыс жүктемесінің қажеттіліктеріне қарай автоматты түрде реттеледі. Бұл механизм есептеу қуаттарына кететін шығынды 60%-ға дейін үнемдеуге мүмкіндік береді.

## {heading(Kubernetes кластерін масштабтау туралы хабарландыру)[id=k8s-scale-alerting]}

Cloud Alerting сервисі кластер түйіндерін масштабтауға байланысты келесі оқиғалар бойынша {linkto(../../../../monitoring-services/alerting/instructions/triggers#alerting-triggers-add)[text=хабарландыруды баптауға]} мүмкіндік береді:

- `kubernetes_master_vscale` — кластердің master-түйінін тік масштабтау;
- `kubernetes_worker_vscale` — түйіндер тобын тік масштабтау;
- `kubernetes_worker_hscale` — түйіндер тобын көлденең масштабтау.

Хабарландырулар уақтылы келуі үшін іске қосылу шарты `больше 0`, ал салыстыру аралығы `1 минута` болуы керек.

## {heading(Қолданыстағы шектеулер)[id=k8s-scale-limitations]}

- Тік масштабтау мүмкіндіктері {linkto(../../../../tools-for-using-services/account/concepts/quotasandlimits#quotasandlimits-quotas)[text=қолданыстағы квоталармен]} және қолжетімді {linkto(../flavors#k8s-flavors-vm-flavor)[text=виртуалды машина қалыптарымен]} шектеледі.
- Көлденең масштабтау мүмкіндіктері қолданыстағы квоталармен және жекелеген түйіндер тобындағы worker-түйіндер санының шегімен шектеледі: 1-ден 500 түйінге дейін.
- Егер автоматты масштабтау бапталған болса, көлденең масштабтауды қолмен орындау мүмкін емес. Масштабтауды қолмен орындау үшін {linkto(../../instructions/scale#k8s-instructions-scale-horizontal-worker-nodes)[text=автоматты масштабтауды өшіріңіз]}.

## {heading(Тік масштабтау кезінде қолжетімділікті қамтамасыз ету)[id=k8s-scale-availability-with-scaling]}

Кластердің кез келген түйінін тік масштабтау кезінде түйіндер пайдаланатын виртуалды машиналар бірізді түрде қайта іске қосылады. Бұл виртуалды машинаның жаңа қалыбын қолдану үшін қажет. Сондықтан масштабтау процесі кластер API-інің де, кластерде орналастырылған жұмыс жүктемелерінің де қолжетімділігіне әсер етеді:

- Егер кластер {linkto(../architecture#k8s-architecture-topology)[text=істен шығуға төзімді]} болмаса және бір ғана master-түйіннен тұрса, онда Kubernetes API масштабтау аяқталғанға дейін қолжетімсіз болады.
- Егер түйіндер тобында бір ғана worker-түйін болса, онда онда орналастырылған жұмыс жүктемесі масштабтау аяқталғанға дейін қолжетімсіз болады.
- Егер түйіндер тобында бірнеше worker-түйін болса, онда оларда орналастырылған жұмыс жүктемесі жүктеме үшін репликация бапталмаған немесе дұрыс бапталмаған жағдайда масштабтау аяқталғанға дейін қолжетімсіз болады.

  Мысалы, егер барлық репликаларды бір worker-түйінде орналастырсаңыз, ол қайта іске қосылғанда жұмыс жүктемесі қолжетімсіз болады, тіпті түйіндер тобында басқа worker-түйіндер болса да.

  Жұмыс жүктемесі репликаларының бір бөлігі worker-түйіндер қайта іске қосылған кезде қолжетімді болатындай етіп репликацияны баптаңыз.

- Егер түйіндер тобын масштабтағаннан кейін жұмыс жүктемесіне есептеу ресурстары жеткіліксіз болса, онда бұл жүктеме дұрыс жұмыс істемеуі немесе қолжетімсіз болуы мүмкін.

  Масштабты азайту кезінде түйіндер тобындағы қорытынды ресурстар көлемі жеткілікті болуын қадағалаңыз.
