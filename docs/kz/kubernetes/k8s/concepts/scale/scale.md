Масштабтау кластерді жұмыс жүктемелерінің ([workloads](https://kubernetes.io/docs/concepts/workloads/)) өзгеретін қажеттіліктеріне бейімдеуге мүмкіндік береді.

## Масштабтау түрлері

Масштабтаудың екі түрі қолданылады:

- _Тік масштабтау_: кластердің master- және worker-түйіндері пайдаланатын [виртуалды машина қалыптары](../flavors) өзгереді. Осылайша түйіндер санына әсер етпей, кластердің есептеу ресурстарын басқаруғал болады.

  Worker-түйіндерді масштабтау [worker-түйіндер тобы](../architecture#cluster_topology) аясында [Cluster Autoscaler](/kz/kubernetes/k8s/concepts/cluster-autoscaler) көмегімен орындалады.

- _Көлденең масштабтау_: жекелеген түйіндер тобы аясындағы worker-түйіндер саны өзгереді. Осылайша түйіндер пайдаланатын виртуалды машина қалыптарына әсер етпей, кластердің есептеу ресурстарын басқаруғал болады.

Екі түрдегі масштабтауды да қолмен орындауғал болады, сондай-алқ [автоматты масштабтау](#autoscaling) жұмыс істейді.

## {heading(Автоматты масштабтау)[id=autoscaling]}

### Тік масштабтау

Master-түйіндердің _тік автоматты масштабталуы_ барлық кластерлер үшін қолданылады, оны өшіруге болмайды.

Автоматты масштабтау агенті CPU және RAM бойынша master-түйін жүктемесін бағалап, келесі шектік мәндерді қадағалайды:

|===

|Шектік мән
|Автомасштабтаудың әрекеті

|CPU жүктемесі 60 секунд бойы 80%-дан асады
.2+|Егер шектердің кем дегенде біреуі асып кетсе, [Cloud Containers](/kz/kubernetes/k8s) сервисіне master-түйіннің ВМ шаблонын өзгертуге сұрау жіберіледі:

- CPU буыны өзгермейді.
- CPU ядроларының саны екіге артады.
- RAM көлемі тек CPU санын арттыру үшін қажет болған жағдайда ғана өзгереді.

- `STD2-2-6` → `STD2-4-6`.
- `STD2-6-6` → `STD2-8-8`.
- `STD3-2-6` → `STD3-4-6`.

- CPU буыны өзгермейді.
- CPU ядроларының саны тек RAM көлемін арттыру үшін қажет болған жағдайда ғана өзгереді.
- RAM көлемі келесі ретпен бір қадамғал өзгереді: 6, 8, 12, 16, 20, 24, 32, 36, 48.

- `STD2-2-6` → `STD2-2-8`.

Master-түйіннің ВМ қалыбын CPU мен RAM мөлшері аз қалыпқал тек [қолмен](../../instructions/scale#scale_master_nodes) және тек [бірінші буын](/kz/kubernetes/k8s/concepts/cluster-generations) кластерлері үшін өзгертуге болады.

{cut(06.12.2024 дейін жасалған кластерлер үшін тік масштабтау сұлбасы)}

Автоматты масштабтау агенті CPU және RAM бойынша master-түйін жүктемесін бағалап, келесі шектік мәндерді қадағалайды:

- CPU жүктемесі 60 секунд бойы 80%-дан асады;
- CPU жүктемесі 5 минут бойы 60%-дан асады;
- RAM жүктемесі 60 секунд бойы 90%-дан асады.

Егер шектердің кем дегенде біреуі асып кетсе, [Cloud Containers](/kz/kubernetes/k8s) сервисіне master-түйіннің ВМ шаблонын өзгертуге сұрау жіберіледі. Бұл ретте CPU және RAM мәндері екі есе ұлғайтылады. Мысалы, `STD2-2-6` шаблоны `STD2-4-12` шаблонына өзгертіледі.

{/cut}

### Көлденең масштабтау

Түйіндер тобы ([autoscaling](/kz/kubernetes/k8s/concepts/cluster-autoscaler)) үшін _көлденең автоматты масштабтау_ пайдаланушы арқылы [бапталады](../../instructions/scale#autoscale_worker_nodes). Топтағы worker-түйіндер саны жұмыс жүктемесінің қажеттіліктеріне қарай автоматты түрде реттеледі. Бұл механизм есептеу қуаттарына кететін шығынды 60%-ғал дейін үнемдеуге мүмкіндік береді.

## {heading(Kubernetes кластерін масштабтау туралы хабарландыру)[id=alerting]}

Cloud Alerting сервисі кластер түйіндерін масштабтауғал байланысты келесі оқжәнеғалар бойынша [хабарландыруды баптауғал](/kz/monitoring-services/alerting/instructions/triggers#triggers_add) мүмкіндік береді:

- `kubernetes_master_vscale` — кластердің master-түйінін тік масштабтау;
- `kubernetes_worker_vscale` — түйіндер тобын тік масштабтау;
- `kubernetes_worker_hscale` — түйіндер тобын көлденең масштабтау.

Хабарландырулар уақтылы келуі үшін іске қосылу шарты `больше 0`, ал салыстыру аралығы `1 минута` болуы керек.

## Қолданыстағы шектеулер

- Тік масштабтау мүмкіндіктері [қолданыстағы квоталармен](/kz/tools-for-using-services/account/concepts/quotasandlimits#kvotalar) және қолжетімді [виртуалды машина қалыптарымен](../flavors#konfiguraciya_kalyptary) шектеледі.
- Көлденең масштабтау мүмкіндіктері қолданыстағы квоталармен және жекелеген түйіндер тобындағы worker-түйіндер санының шегімен шектеледі: 1-ден 500 түйінге дейін.
- Егер автоматты масштабтау бапталған болса, көлденең масштабтауды қолмен орындау мүмкін емес. Масштабтауды қолмен орындау үшін [автоматты масштабтауды өшіріңіз](../../instructions/scale#worker_tuyinder_toptaryn_masshtabtau_b32256e5).

## Тік масштабтау кезінде қолжетімділікті қамтамасыз ету

Кластердің кез келген түйінін тік масштабтау кезінде түйіндер пайдаланатын виртуалды машиналар бірізді түрде қайта іске қосылады. Бұл виртуалды машинаның жаңал қалыбын қолдану үшін қажет. Сондықтан масштабтау процесі кластер API-інің де, кластерде орналастырылған жұмыс жүктемелерінің де қолжетімділігіне әсер етеді:

- Егер кластер [істен шығуғал төзімді](../architecture#cluster_topology) болмаса және бір ғана master-түйіннен тұрса, онда Kubernetes API масштабтау аяқталғанғал дейін қолжетімсіз болады.
- Егер түйіндер тобында бір ғана worker-түйін болса, онда онда орналастырылған жұмыс жүктемесі масштабтау аяқталғанғал дейін қолжетімсіз болады.
- Егер түйіндер тобында бірнеше worker-түйін болса, онда оларда орналастырылған жұмыс жүктемесі жүктеме үшін репликация бапталмаған немесе дұрыс бапталмаған жағдайда масштабтау аяқталғанғал дейін қолжетімсіз болады.

  Мысалы, егер барлық репликаларды бір worker-түйінде орналастырсаңыз, ол қайта іске қосылғанда жұмыс жүктемесі қолжетімсіз болады, тіпті түйіндер тобында басқал worker-түйіндер болса да.

  Жұмыс жүктемесі репликаларының бір бөлігі worker-түйіндер қайта іске қосылған кезде қолжетімді болатындай етіп репликацияны баптаңыз.

- Егер түйіндер тобын масштабтағаннан кейін жұмыс жүктемесіне есептеу ресурстары жеткіліксіз болса, онда бұл жүктеме дұрыс жұмыс істемеуі немесе қолжетімсіз болуы мүмкін.

  Масштабты азайту кезінде түйіндер тобындағы қорытынды ресурстар көлемі жеткілікті болуын қадағалаңыз.
