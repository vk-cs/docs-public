## Кластерішілік DNS-сервер

Кластер ішіндегі [CoreDNS](https://coredns.io/) сервері `kube-dns` орнына пайдаланылады. Бұл сервер кластерде [сервистерді анықтауды](https://coredns.io/plugins/kubernetes/) қамтамасыз етеді және оларғал DNS атаулары арқылы жүгінуге мүмкіндік береді.

Сондай-алқ CoreDNS [метрикаларды](https://coredns.io/plugins/metrics/) Prometheus-қал экспорттайды, бұл оның жұмысын кластер мониторингі құралдары арқылы бақылауғал мүмкіндік береді.

## Container Network Interface (CNI)-мен жұмыс

Kubernetes кластерлері кластерішілік желіні ұйымдастыру үшін [Calico](https://projectcalico.docs.tigera.io/about/about-calico) пайдаланады. Calico VK Cloud платформасымен меншікті әзірлемедегі бағдарламалық анықталатын [SDN Sprut](/kz/networks/vnet/concepts/sdn#sprut) желісі арқылы өзара әрекеттеседі. SDN Sprut-қал қол жеткізу алу үшін [техникалық қолдау қызметіне](/kz/contacts) хабарласыңыз.

## Жүктеме теңгергіштерімен интеграция

Kubernetes кластеры VK Cloud платформасының жүктеме теңгергіштерімен интеграцияланады. Бұл Kubernetes кәдімгі жүктеме теңгергіштеріне (`LoadBalancer`) де, Ingress-контроллерлеріне (`IngressController`) де қатысты: екеуін жасаған кезде де оларғал VK Cloud-тың бөлінген TCP-теңгергіші байланыстырылатын болады. Бұл соның ішінде [аддон](../addons-and-settings/addons) түрінде орнатылатын Ingress-контроллеріне де қатысты.

Қажет болған жағдайда HTTP жүктеме теңгергішін пайдалануғал болады. Бұл туралы толығырақ [Ingress-контроллерге арналған мысалда](../../how-to-guides/ingress/ingress-http) айтылған.

VK Cloud платформасының жүктеме теңгергіші [OpenStack Octavia](https://docs.openstack.org/octavia/latest/) негізінде құрылған, ал оның негізінде [HAProxy](http://www.haproxy.org/) жатыр және ол мыналарды қолдайды:

- HTTP және TCP қосылыстарын проксилеу мен теңгерімдеу (соңғылары — соның ішінде proxy протоколын қолдаумен);
- HTTP/1.1-ге қоса HTTP/2 қосылыстарын проксилеу мен теңгерімдеу;
- теңгергіште SSL қосылыстарын терминирлеу.

Істен шығуғал төзімділік үшін теңгергіштің екі данасы қолдау көрсетіледі, олардың біреуі active режимінде, ал екіншісі standby режимінде болады. Осы теңгергіштер арасындағы күйді синхрондау мен трафикті ауыстыру [VRRP](https://www.rfc-editor.org/rfc/rfc5798) протоколының көмегімен орындалады. Кластер мұндай істен шығуғал төзімді конфигурацияны бір теңгергіш ретінде қабылдайды.

## Ingress-контроллер және пайдаланушының нақты IP мекенжайын анықтау

Кейде Ingress-контроллерді пайдаланған кезде кластердегі подқал теңгергішке бастапқы сұрауды жіберген пайдаланушының нақты IP мекенжайын көру қажет болады, ал теңгергіштің өзінің IP мекенжайын емес. Бұл кейбір желілік қолданбалар түрлері үшін маңызды болуы мүмкін.

Ingress-контроллер арқылы жұмыс істейтін под пайдаланушының нақты IP мекенжайын көре алуы үшін, келесі нұсқалардың бірін пайдаланыңыз:

- [proxy протоколын қолдайтын Ingress-контроллер](../../how-to-guides/ingress/ingress-tcp).

  Егер HTTPS трафигімен жұмыс істеу жоспарланса, осы Ingress-контроллерде SSL қосылыстарын терминирлеуді баптаңыз, себебі контроллер үшін жасалатын TCP-теңгергіш SSL қосылыстарын өзі терминирлей алмайды.

  VK Cloud ұсынатын [NGINX негізіндегі Ingress-контроллер](../addons-and-settings/addons) proxy протоколын қолдайды және онымен жұмыс істеуге әлдеқашан бапталған.

- [Қосымша баптаулары бар жеке HTTP\HTTPS жүктеме теңгергіші](../../how-to-guides/ingress/ingress-http):

  - Егер HTTPS трафигімен жұмыс істеу жоспарланса, осы теңгергіште SSL қосылыстарын терминирлеуді баптаңыз.
  - Ingress-контроллерде `ExternalTrafficPolicy: Local` саясатын белсендіріңіз.

  Бұл жағдайда Ingress-контроллерге тек HTTP трафигі түседі, онда барлық тақырыптар, соның ішінде жіберушінің IP мекенжайы да (егер бар болса) көрінетін болады.

## Әдепкі желілік нысандар

Kubernetes кластерін жасаған кезде ол үшін бірнеше желілік нысан жасалады.

### Kubernetes API үшін жүктеме теңгергіші

Әр кластер үшін барлық master-түйіндерге арналған Kubernetes API-іне келетін кіріс сұрауларды өңдейтін бөлінген TCP жүктеме теңгергіші жасалады. Кластерге қосылу және оны басқару осы теңгергіш арқылы жүзеге асады.

### Қауіпсіздік топтарының ережелері

Әр кластер үшін үш ереже тобы жасалады:

- `<имя кластера>-base`: master-түйіндері мен worker-түйіндері топтары арасындағы байланысты қамтамасыз етеді.
- `<имя кластера>-master`: master-түйіндермен байланысты қамтамасыз етеді.
- `<имя кластера>-minion`: worker-түйіндері топтары арасындағы байланысты қамтамасыз етеді.

{note:warn}

Бұл ереже топтарын өзгерту кластердің жұмысқал жарамсыз болуына әкелуі мүмкін.

{/note}
