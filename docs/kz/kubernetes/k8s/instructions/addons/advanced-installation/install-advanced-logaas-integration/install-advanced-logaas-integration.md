## Бірінші буындағы кластерлерде Cloud Logging (logaas-integration) үшін Fluent Bit

Жобада [Cloud Logging](/kz/monitoring-services/logging) сервисін қосыңыз, егер бұл әлі жасалмаған болса. Ол үшін [техникалық қолдау қызметіне хабарласыңыз](/kz/contacts).

## Дайындық қадамдары

Аддон үшін [орнатудың бірнеше нұсқасы](../../../../concepts/addons-and-settings/addons#addondardy_ornatu_erekshelikteri) қолжетімді:

- стандартты орнату;
- жылдам орнату.

Таңдалған орнату нұсқасына қарамастан, аддон кластердің барлық тораптарына, соның ішінде master-тораптарғал [DaemonSet](https://kubernetes.io/docs/concepts/workloads/controllers/daemonset/) түрінде орнатылады.

worker-тораптар топтарына орналастырылатын аддондардың [жиынтық ең жоғары жүйелік талаптарын](../../../../concepts/addons-and-settings/addons) ескеріңіз. Қажет болса, орнату алдында worker-тораптар топтарын [қолмен масштабтауды орындаңыз](../../../scale#scale_worker_nodes) немесе [автоматты масштабтауды баптаңыз](../../../scale#autoscale_worker_nodes).

## Орнату кезінде аддонды баптау кодын өңдеу

Аддон кодын өңдеу стандартты орнатуғал қолданылады.

Өрістердің сипаттамасымен бірге аддонды баптаудың толық коды мына жерде қолжетімді:

- жеке кабинетте;
- егер Terraform пайдаланылса, `configuration_values` деректер көзіндегі [configuration_values](https://github.com/vk-cs/terraform-provider-vkcs/blob/master/docs/data-sources/kubernetes_addon.md) атрибутында.

Сондай-алқ [GitHub](https://github.com/fluent/helm-charts/blob/main/charts/fluent-bit/values.yaml) сайтында осы аддонның негізі болып табылатын Fluent Bit баптау коды қолжетімді.

{note:warn}

Аддонды дұрыс орнату және оның жұмыс істеуі үшін қажет өрістерді немесе осы өрістерде берілген мәндерді жоймаңыз.

Аддонды баптау кодында осындай өрістерді табуғал мүмкіндік беретін түсіндірмелер бар.

{/note}

[pipeline](https://docs.fluentbit.io/manual/pipeline) пайплайны және [конфигурация файлының баптаулары](https://docs.fluentbit.io/manual/administration/configuring-fluent-bit/classic-mode/configuration-file) туралы толығырақ Fluent Bit ресми құжаттамасынан оқыңыз.

### Әртүрлі критикалық деңгейлермен жұмыс істеу кезінде аддонның мінез-құлқын дәл баптау

Логтарды Cloud Logging сервисіне жіберер алдында аддон келесі әрекеттерді орындайды:

1. Логтардағы жекелеген жазбалардың критикалық деңгейін (severity level) анықтайды. Бұл Fluent Bit парсерлері ([parsers](https://docs.fluentbit.io/manual/pipeline/filters/parser)) арқылы жасалады.

   {cut(Қолдау көрсетілетін деңгейлер кестесі)}

   Деңгейлер критикалылықтың өсу ретімен сұрыпталған.

   <!-- prettier-ignore -->
   | Деңгей | Сандық баламасы | Анықтамасы                      |
   | ------- | ------------------- | ------------------------------- |
   | `DEBUG` | 4                   | Жөндеу хабарламалары            |
   | `INFO`  | 3                   | Ақпараттық хабарламалар         |
   | `WARN`  | 2                   | Ескерту хабарламалары           |
   | `ERROR` | 1                   | Қате туралы хабарламалар        |
   | `FATAL` | 0                   | Критикалық қателер туралы хабарламалар |

   {/cut}

1. Логтарғал қосымша метадеректерді қосады, бұл Cloud Containers кластерлерінің логтарымен жұмысты жеңілдетеді (мысалы, Cloud Logging ішінде қажетті логтарды іздеуді). Ол үшін Lua тілінде [жазылған](https://docs.fluentbit.io/manual/pipeline/filters/lua) арнайы Fluent Bit сүзгілері (filters) қолданылады. Бұл метадеректер, соның ішінде, журналғал жазылатын оқжәнеғалардың критикалық деңгейін қамтиды.

   {cut(Логтар көзін анықтауға арналған метадеректер туралы толығырақ)}

   Логтардағы әрбір жазба үшін Cloud Logging пайдаланған кезде логтар көзін анықтауғал мүмкіндік беретін идентификаторлар беріледі:

   - Ресурстар тобының идентификаторы (`group-id`): кластер атауы.
   - Ресурс идентификаторы (`stream-id`) келесі пішімдердің бірінде:

     - [kubelet қызметі](https://kubernetes.io/docs/concepts/overview/components/#kubelet) үшін: `<имя узла кластера (node)>.<полное имя службы>`.

       **Мысал:**

       ```text
       my-cluster-node-0.kubelet.service
       ```

     - [под](../../../../reference/pods) (pod) үшін: `<имя пространства имен (namespace)>.<имя>`.

       Алдымен атау келесі белгілерден (labels) ізделеді:

       - `app.kubernetes.io/instance` ([белгі туралы толығырақ](https://kubernetes.io/docs/reference/labels-annotations-taints/#app-kubernetes-io-instance));
       - `app.kubernetes.io/name` ([белгі туралы толығырақ](https://kubernetes.io/docs/reference/labels-annotations-taints/#app-kubernetes-io-name));
       - `app`;
       - `k8s-app`.

       Егер мұндай белгілер болмаса, подтың атауы пайдаланылады.

       **Мысал:**

       Айталық:

       - `kube-system` атаулар кеңістігінде `kube-controller-manager-my-cluster-master-0` атауы бар под бар.
       - Осы под үшін `k8s-app=kube-controller-manager` белгісі берілген.

       Онда ресурс идентификаторы мынадай болады:

       ```text
       kube-system.kube-controller-manager
       ```

   {/cut}

   {cut(Логтар туралы қосымша ақпараты бар басқа метадеректер туралы толығырақ)}

   Логтардағы жазбаларғал журналғал жазылатын оқжәнеғал туралы қосымша алқпараты бар метадеректер өрістері қосылады, бұл Cloud Logging ішінде іздеуді лог метадеректері бойынша да орындауғал мүмкіндік береді:

   - `node_name`: под немесе kubelet қызметі орналасқан кластер торабының атауы.
   - `severity`: журналғал жазылатын оқжәнеғаның критикалық деңгейі (severity level) мәтіндік түрде.
   - `severity_num`: журналғал жазылатын оқжәнеғаның критикалық деңгейі (severity level) сандық түрде.
   - JSON-хабарламасының барлық өрістері: егер логтар JSON пішімінде жазылса, `msg` өрісінен басқал JSON-хабарламасының барлық өрістері метадеректер түрінде қосылады.

   {/cut}

Критикалық деңгейлермен жұмыс істеу кезінде аддонның мінез-құлқын дәл баптауды келесілерді пайдаланып орындай аласыз:

{tabs}

{tab(Пайдаланушы сүзгілері)}

Cloud Logging ішіне тек берілген ең төменгі критикалық деңгейі бар логтар түсуі үшін, аддон кодында `customFilter` пайдаланушы сүзгілері үшін бір немесе бірнеше ережені беріңіз. Бұл ережелерді белгілі бір атаулар кеңістігі деңгейінде және атаулар кеңістігіндегі белгілі бір подтар деңгейінде баптауғал болады:

<!-- prettier-ignore -->
```yaml
customFilter:
  - namespace: <имя пространства имен>
    rules: # Одно правило для пространства имен
      - min_level: <буквенное обозначение минимального уровня критичности>
  - namespace: <имя другого пространства имен>
    rules: # Несколько правил для подов в пространстве имен
      - podprefix: <префикс имени пода>
        min_level: <буквенное обозначение минимального уровня критичности>
      - podprefix: <префикс имени другого пода>
        min_level: <буквенное обозначение минимального уровня критичности>
  - namespace: <имя другого пространства имен>
    rules: # Комбинация правила для пространства имен и правил для подов
      - min_level: <буквенное обозначение минимального уровня критичности>
      - podprefix: <префикс имени пода>
        min_level: <буквенное обозначение минимального уровня критичности>
      - podprefix: <префикс имени пода>
        min_level: <буквенное обозначение минимального уровня критичности>
```

Бір жұмыс жүктемесіне ([workload](https://kubernetes.io/docs/concepts/workloads/controllers/)) жататын бірнеше реплика-подтан логтар алуғал болу үшін, подтар үшін дәл префикс бапталады.

{cut(Мысалдар)}

- Жалғыз ереже `kube-system` атаулар кеңістігі деңгейінде қолданылады.

   Cloud Logging ішіне осы атаулар кеңістігінен тек `WARN` немесе одан жоғары (`ERROR`, `FATAL`) критикалық деңгейі бар логтар түседі.

- Жалғыз ереже `default` атаулар кеңістігінде орналасқан және атаулары `test-pod` префиксінен басталатын подтар деңгейінде қолданылады.

   Cloud Logging ішіне осы подтардан тек `ERROR` немесе одан жоғары (`FATAL`) критикалық деңгейі бар логтар түседі. `default` атаулар кеңістігі деңгейінде логтарды сүзгілеу жүргізілмейді.

- Бірнеше ереже `my-namespace` атаулар кеңістігі деңгейінде және осы атаулар кеңістігінде орналасқан `example-pod` префиксі бар подтар деңгейінде қолданылады.

   Cloud Logging ішіне тек келесі логтар түседі:

   - `example-pod` префиксі бар подтарғал жатпайтын және `ERROR` немесе одан жоғары (`FATAL`) критикалық деңгейі бар атаулар кеңістігіндегі логтар.
   - `example-pod` префиксі бар подтардан `WARN` немесе одан жоғары (`ERROR`, `FATAL`) критикалық деңгейі бар логтар.

<!-- prettier-ignore -->
```yaml
customFilter:
  - namespace: kube-system
    rules:
      - min_level: WARN
  - namespace: default
    rules:
      - podprefix: test-pod
        min_level: ERROR
  - namespace: my-namespace
    rules:
      - min_level: ERROR
      - podprefix: example-pod
        min_level: WARN
```

{/cut}

{tab(Пайдаланушы үлгілері)}

Стандартты аддон механизмдерін айналып өтіп, логтар үшін қажетті критикалық деңгейді орнату үшін, аддон кодында `customRegexp` пайдаланушы үлгілері үшін бір немесе бірнеше ережені беріңіз. Егер логтағы жазбаның бір бөлігі берілген Lua үлгісімен ([pattern](https://www.lua.org/manual/5.4/manual.html#6.4.1)) сәйкес келсе, онда осы жазбағал ережеде берілген критикалық деңгей тағайындалады. Егер сәйкестіктер болмаса, жазбағал Fluent Bit парсерлері арқылы анықталған критикалық деңгей тағайындалады. Бұл аддонда қолданылатын деңгейді автоматты түрде анықтау механизмі белгілі бір қолданбаның логтар деңгейін қате анықтайтын болса, пайдалы болуы мүмкін.

Бұл ережелер белгілі бір атаулар кеңістігі деңгейінде және атаулар кеңістігіндегі белгілі бір подтар деңгейінде әрекет етеді:

<!-- prettier-ignore -->
```yaml
customRegexp:
  - namespace: <имя пространства имен>
    rules:
    - levels: # Одно правило для пространства имен: настройка нескольких уровней
      - level: <буквенное обозначение желаемого уровня критичности>
        reg_exp: "<шаблон Lua>"
      - level: <буквенное обозначение другого желаемого уровня критичности>
        reg_exp: "<шаблон Lua>"
  - namespace: <имя другого пространства имен>
    rules: # Несколько правил для подов в пространстве имен: настройка нескольких уровней
    - podprefix: <префикс имени пода> # Первое правило для подов
      levels:
      - level: <буквенное обозначение желаемого уровня критичности>
        reg_exp: "<шаблон Lua>"
      - level: <буквенное обозначение другого желаемого уровня критичности>
        reg_exp: "<шаблон Lua>"
    - podprefix: <префикс имени другого пода> # Второе правило для подов
      levels:
      - level: <буквенное обозначение желаемого уровня критичности>
        reg_exp: "<шаблон Lua>"
  - namespace: <имя другого пространства имен>
    rules:
    - levels: # Первое правило для пространства имен: настройка нескольких уровней
      - level: <буквенное обозначение желаемого уровня критичности>
        reg_exp: "<шаблон Lua>"
      - level: <буквенное обозначение другого желаемого уровня критичности>
        reg_exp: "<шаблон Lua>"
    - podprefix: <префикс имени пода> # Второе правило для подов: настройка нескольких уровней
      levels:
      - level: <буквенное обозначение желаемого уровня критичности>
        reg_exp: "<шаблон Lua>"
      - level: <буквенное обозначение другого желаемого уровня критичности>
        reg_exp: "<шаблон Lua>"
```

Бір жұмыс жүктемесіне ([workload](https://kubernetes.io/docs/concepts/workloads/controllers/)) жататын бірнеше реплика-подтан логтар алуғал болу үшін, подтар үшін дәл префикс бапталады.

{cut(Мысалдар)}

- Жалғыз ереже `kube-system` атаулар кеңістігі деңгейінде қолданылады.

   Егер осы атаулар кеңістігіндегі лог жазбасының бір бөлігі `[Ww]%d%d%d%d%s+` берілген үлгісімен сәйкес келсе, онда осы жазбағал `ERROR` критикалық деңгейі тағайындалады.

- Жалғыз ереже `default` атаулар кеңістігінде орналасқан және атаулары `test-pod` префиксінен басталатын подтар деңгейінде қолданылады.

   Егер осы подтардағы лог жазбасының бір бөлігі `this is a plain text message` берілген үлгісімен сәйкес келсе, онда осы жазбағал `WARN` критикалық деңгейі тағайындалады.

- Бірнеше ереже `my-namespace` атаулар кеңістігі деңгейінде және осы атаулар кеңістігінде орналасқан `example-pod` префиксі бар подтар деңгейінде қолданылады.

   Критикалық деңгейлер келесі ережелер бойынша тағайындалады:

   - Егер осы атаулар кеңістігіндегі лог жазбасының (`example-pod` префиксі бар подтарғал жатпайтын) бір бөлігі `[Ww]%s+` берілген үлгісімен сәйкес келсе, онда осы жазбағал `ERROR` критикалық деңгейі тағайындалады.
   - Егер `example-pod` префиксі бар подтардағы лог жазбасының бір бөлігі `Debug trace` берілген үлгісімен сәйкес келсе, онда осы жазбағал `DEBUG` критикалық деңгейі тағайындалады.

Егер лог жазбасының ешбір бөлігі көрсетілген үлгілермен сәйкес келмесе, онда мұндай жазбағал Fluent Bit парсерлері арқылы анықталған критикалық деңгей тағайындалады.

<!-- prettier-ignore -->
```yaml
customRegexp:
  - namespace: kube-system
    rules:
    - levels:
      - level: ERROR
        reg_exp: "[Ww]%d%d%d%d%s+"
  - namespace: default
    rules:
    - podprefix: test-pod
      levels:
      - level: WARN
        reg_exp: "this is a plain text message"
  - namespace: my-namespace
    rules:
    - levels:
      - level: ERROR
        reg_exp: "[Ww]%s+"
    - podprefix: example-pod
      levels:
      - level: DEBUG
        reg_exp: "Debug trace"
```
